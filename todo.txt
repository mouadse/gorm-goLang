PHASE 3: WORKOUT DOMAIN EXPANSION
================================================================================

[ ] 6. Cardio Entries (First-Class Model)
    Schema:
      - workout_cardio_entries table
      Fields: id, workout_id, modality, duration_minutes, distance, distance_unit,
              pace, calories_burned, avg_heart_rate (optional), notes, created_at, updated_at
    Endpoints:
      - GET /v1/workouts/{id}/cardio
      - POST /v1/workouts/{id}/cardio
      - PATCH /v1/workout-cardio/{id}
      - DELETE /v1/workout-cardio/{id}
    Business Rules:
      - Support running, cycling, walking, swimming, rowing, generic cardio
      - Allow workout to contain both resistance exercises AND cardio entries
    Tests:
      - Mixed resistance + cardio workout
      - Invalid negative values
      - Derived calorie adjustment compatibility later

[ ] 7. Workout Templates
    Schema:
      - workout_templates table (id, owner_id, name, type, notes, created_at, updated_at)
      - workout_template_exercises table (id, template_id, exercise_id, order, sets, reps, weight, rest_time)
      - workout_template_sets table (id, template_exercise_id, set_number, reps, weight, rest_seconds)
    Endpoints:
      - POST /v1/workout-templates
      - GET /v1/workout-templates
      - GET /v1/workout-templates/{id}
      - PATCH /v1/workout-templates/{id}
      - DELETE /v1/workout-templates/{id}
      - POST /v1/workout-templates/{id}/apply
    Business Rules:
      - Applying a template creates a normal workout snapshot
      - Editing a template never mutates existing workouts
    Tests:
      - Template apply creates correct workout hierarchy
      - Ordering is preserved
      - Auth scoping

[ ] 8. Workout Programs (Admin)
    Schema:
      - workout_programs table (id, name, description, created_by, is_active, created_at, updated_at)
      - program_weeks table (id, program_id, week_number, name)
      - program_sessions table (id, week_id, day_number, workout_template_id or inline exercises)
      - program_assignments table (id, user_id, program_id, assigned_at, started_at, completed_at, status)
    Endpoints:
      - Admin CRUD for /v1/admin/programs
      - POST /v1/programs/{id}/assign (admin only)
      - GET /v1/users/{user_id}/program
      - GET /v1/programs/{id}/progress
    Business Rules:
      - Admins create programs
      - Programs are assignable to members
      - Assigned program content should be snapshotted
      - Avoid mutable references that rewrite historical plan
    Tests:
      - Admin-only access
      - Assignment lifecycle
      - Historical snapshot safety

[ ] 15. Workout-Nutrition Integration Engine
    Rules from Scope:
      - Leg day logged -> +200 kcal (mostly carbs)
      - No workout today -> -200 kcal, protein target maintained
      - High volume week (20+ sets/muscle) -> +15% calories
      - 30 min cardio logged -> +300 kcal bonus
      - Recovery warning when training load high AND protein too low
      - Goal-alignment warning when intake conflicts with stated goal
    Implementation:
      - Central integration/rules service
      - Responses expose: final adjusted targets + explanation of which rules fired
    Endpoints:
      - Extend summary endpoints
      - GET /v1/users/{user_id}/recommendations
    Tests:
      - One test per rule
      - Combined rule precedence
      - No silent adjustments without explanation payload
    Files:
      - services/integration_rules.go

[ ] 16. Exports & GDPR Workflows
    Schema:
      - exports table (id, user_id, format, status, file_path, created_at, completed_at)
      - optional: deletion_requests table
    Endpoints:
      - POST /v1/exports (create export job)

    Export Formats:

      - CSV second
    Export Scope:
      - User profile
      - Workouts, workout exercises, workout sets
      - Cardio entries
      - Meals, meal foods
      - Foods, recipes
      - Weight entries
      - Settings and targets
    Tests:
      - Export job creation
      - Export package completeness
      - Ownership and auth checks

[ ] 17. Notifications
    Schema:
      - notifications table (id, user_id, type, title, message, payload_json, read_at, created_at)
    Endpoints:
      - GET /v1/notifications
      - PATCH /v1/notifications/{id}/read
    Trigger Sources:
      - Low protein warning
      - Missed meal logging
      - Workout reminder
      - Rest-day warning
      - Export ready
    Realtime Delivery:
      - WebSocket or SSE can come after persistence
    Tests:
      - Notification creation from rule outputs
      - Mark-read flow
      - No duplicate spam for same trigger window

[ ] 18. Two-Factor Authentication (2FA)
    Schema:
      - two_factor_secrets table (id, user_id, secret, verified, created_at)
      - recovery_codes table (id, user_id, code_hash, used_at, created_at)
    Endpoints:
      - POST /v1/auth/2fa/setup
      - POST /v1/auth/2fa/verify
      - POST /v1/auth/2fa/disable
      - POST /v1/auth/2fa/recover (use recovery code)
    Business Rules:
      - TOTP-based flow (Google Authenticator compatible)
      - Recovery code fallback (generate 8-10 codes on setup)
      - Login challenge after password verification
    Tests:
      - Valid TOTP
      - Invalid TOTP
      - Recovery code consumption
      - Disable flow
